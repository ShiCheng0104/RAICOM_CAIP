## 3. Flink 微批量优化方案

v4 之前的 Flink 主链路是逐条同步打分：

```text
event -> enrich -> POST /predict(record) -> wait -> output
```

这种方式实现简单，但每条交易都要等待一次 HTTP 往返，吞吐容易被 API 延迟限制。

v5 新增 `MicroBatchWindowScorer`，将同一个 Flink task 内的多条交易先做画像和窗口特征补齐，再批量调用模型：

```text
event_1 -> enrich -> buffer
event_2 -> enrich -> buffer
...
buffer size >= N 或 linger 超时
    -> POST /predict(records=[...])
    -> 拆分 results
    -> 写入 risk_results / alert_events / late_events
```

为了补齐 demo 版有限流回放时的尾批问题，v5 增加了显式 flush 控制消息：

```text
Producer / Benchmark 有限回放结束
    -> 向每个 Kafka partition 写入 __fraudsim_flush=true
    -> Flink 微批算子收到 flush marker
    -> 立即提交当前 subtask buffer 中尚未满批的数据
```

这样既保留连续流下的 batch size / linger 触发，也保证答辩演示、压测脚本这类有限输入不会遗留尾批。

当前配置：

| 配置项 | 默认值 | 含义 |
|---|---:|---|
| `FRAUDSIM_SCORING_MODE` | `microbatch` | 使用微批量模式 |
| `FRAUDSIM_BATCH_SIZE` | 50 | 每批最多 50 条 |
| `FRAUDSIM_BATCH_LINGER_MS` | 300 | 最多等待 300 ms 后尝试 flush |
| `FRAUDSIM_FLINK_PARALLELISM` | 2 | Flink 并行度 |

相关文件：

| 文件 | 作用 |
|---|---|
| `fraudsim/streaming/flink_job.py` | 新增 `MicroBatchWindowScorer` |
| `docker-compose.yml` | 默认启用 `microbatch` 模式 |
| `fraudsim/streaming/producer.py` | 有限回放结束后发送 flush marker |
| `fraudsim/streaming/benchmark.py` | 压测时发送 flush marker，并修正消费起点 |
| `tests/test_streaming_microbatch.py` | 验证批量 flush 和尾批 flush 行为 |

## 4. API 延迟实测对比

本次使用本地临时 API `http://127.0.0.1:8010`，加载当前 LightGBM 模型 `20260527T040026Z`，使用 200 条样本进行测试。

测试脚本：

```powershell
.\.conda\ruikang\python.exe .\scripts\benchmark_api_latency.py `
  --api-url http://127.0.0.1:8010 `
  --records 200 `
  --batch-size 50 `
  --output models\api_latency_microbatch_report.json
```

测试结果：

| 调用方式 | 请求数 | 平均延迟 | P50 | P95 | P99 |
|---|---:|---:|---:|---:|---:|
| 单笔 `/predict(record)` | 200 | 125.26 ms/条 | 122.95 ms | 138.28 ms | 158.64 ms |
| 批量 `/predict(records)` | 4 批 | 135.92 ms/批 | 138.88 ms | 143.65 ms | 143.65 ms |
| 批量折算单条 | 200 | 2.72 ms/条 | 2.78 ms | 2.87 ms | 2.87 ms |

结论：

```text
单条平均延迟：125.26 ms
微批量折算单条延迟：2.72 ms
平均单条调用层面提升：约 46.08 倍
```

这说明当前模型服务本身已经适合批量推理。Flink 主链路从逐条同步 HTTP 改为算子内微批量后，可以显著降低模型调用摊销成本。

## 5. Docker 内 Flink 端到端压测

在补齐尾批刷新机制后，本次重新使用 Docker 内完整链路进行压测：

```text
本机 benchmark producer
    -> Kafka transaction topic
    -> Docker 内 PyFlink risk job
    -> Redis 画像关联
    -> Docker 内 Model API
    -> Kafka risk topic
    -> benchmark consumer 统计结果
```

测试设置：

| 项目 | 值 |
|---|---|
| 输入样本 | 200 条 |
| 输入速率 | 约 50 条/秒 |
| Flink 并行度 | 2 |
| 微批 batch size | 10 |
| 微批 linger | 300 ms |
| 模型 | LightGBM `20260527T040026Z` |
| flush marker | 开启 |

同口径结果：

| 模式 | sent | received | 完成率 | 平均延迟 | P95 | P99 | 端到端吞吐 |
|---|---:|---:|---:|---:|---:|---:|---:|
| 同步逐条 HTTP | 200 | 200 | 100% | 12.00s | 21.06s | 21.39s | 7.90 条/秒 |
| Flink 微批量 | 200 | 200 | 100% | 3.20s | 4.06s | 4.12s | 28.74 条/秒 |

优化效果：

```text
平均延迟降低：73.31%
P95 延迟降低：80.71%
P99 延迟降低：80.74%
平均延迟加速：约 3.75 倍
P95 延迟加速：约 5.18 倍
```

报告文件：

```text
models/flink_e2e_microbatch_flush_200_rate50_batch10.json
models/flink_e2e_sync_200_rate50_flushmark.json
models/flink_e2e_benchmark_comparison_20260604.json
```

## 6. Dashboard 新增演示能力

v5 新增“演示”页，提供几个答辩可直接点击的按钮：

| 按钮 | 作用 | 背后命令 |
|---|---|---|
| 初始化画像 | 将用户、商户、设备、IP、图统计画像加载到 Redis | `python -m fraudsim.streaming.load_profiles ...` |
| 回放交易流 | 向 Kafka 写入一小段 `transaction_events` | `python -m fraudsim.streaming.producer ...` |
| 轻量重训模型 | 运行 `sklearn_logistic` 轻量重训演示 | `python -m fraudsim.training.train --model sklearn_logistic` |
| 查看实时输出 | 跳转到实时页并拉取 `risk_results` | Dashboard 内部动作 |
| 展示模型对比 | 跳转到指标页并展示多模型条形图 | Dashboard 内部动作 |

这些按钮通过 FastAPI 的本地演示接口触发：

```text
GET  /demo/actions
POST /demo/run
```

注意：这些按钮面向本机比赛演示，不建议在生产环境直接开放命令执行接口。生产环境应改为受权限控制的任务调度服务。

## 7. 图链路可视化

v5 在 Dashboard “实时”页新增图链路面板。点击任意风险事件后，会从事件中抽取：

```text
payer_id
payee_id
merchant_id
device_id
ip_id
risk_score / decision
graph_features
window_features
reason_codes
```

展示为：

```text
用户 -> 收款方
用户 -> 商户
用户 -> 设备
用户 -> IP
商户/设备/IP -> 风险结果
```

答辩时可以这样解释：

> 系统不仅输出风险分，还会把交易涉及的用户、商户、设备、IP、收款方组成一张小型关系图，并同步展示图统计特征和窗口特征。这样人工审核人员可以看到风险来自哪里，例如共享设备、可疑 IP、欺诈邻域或高危商户聚集。

当前图链路是单笔交易局部图，不是完整团伙子图搜索。后续可以继续扩展为从 `graph_edges.parquet` 或图数据库中查询多跳链路。

## 10. 当前边界

v5 已完成微批量 Flink 作业代码、尾批刷新机制、API 延迟实测和 Docker 内 Flink 端到端压测，仍有几个边界：

```text
1. demo 版尾批刷新依赖有限回放结束时写入 flush marker；生产级仍建议使用 Flink processing-time timer 或 Async I/O。
2. Dashboard 的演示命令接口适合本地答辩，不应直接暴露到公网生产环境。
3. 图链路可视化目前是单笔交易局部图，还不是完整多跳团伙图挖掘。
```

## 11. v5 答辩口径

可以这样总结：

```text
v5 重点补强了性能工程化和可解释演示。我们将 Flink 主作业从逐条同步调用扩展为算子内微批量推理，并通过 flush marker 补齐有限流回放的尾批刷新机制。API 层面，单笔平均延迟约 125.26 ms，而 50 条批量推理折算到单条约 2.72 ms，调用层面约 46 倍提升。Docker 内完整 Flink 链路压测中，200 条交易在同步模式下 P95 延迟约 21.06 秒，微批量模式降低到约 4.06 秒，且 200 条全部回收。

同时，Dashboard 新增了演示按钮和图链路可视化。答辩时可以直接点击按钮触发画像加载、交易流回放、轻量重训和模型对比展示；点击风险事件后，可以看到用户、商户、设备、IP 与风险结果之间的链路，解释高风险交易的来源。

因此，系统已经从“可跑通的流式反诈 MVP”进一步增强为“可演示、可解释、可迭代、具备性能优化路径”的比赛作品。
```
