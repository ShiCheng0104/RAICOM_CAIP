# 赛题一解析文档

## 1. 赛题核心目标

赛题一的主题是“欺诈交易智能识别”。核心不是只做一个分类模型，而是围绕金融交易全链路风控场景，完成一套能够模拟交易、实时识别异常、输出风险等级、支持持续迭代，并具备一定可解释性和工程落地能力的系统。

从题目原文来看，最终作品至少要覆盖以下几个关键词：

- 多源异构金融交易数据
- 跨时段、跨渠道交易行为生成
- 正常交易与欺诈交易仿真
- 实时识别与动态异常检测
- 风险评估与风险等级输出
- 多维度特征构建
- 流式数据处理框架
- 自适应学习与模型迭代
- 团伙式欺诈识别
- 可解释性与溯源分析
- 高并发、低延迟、可部署

## 2. 必须完成的任务

### 2.1 搭建仿真交易环境

需要完成的内容：

- 设计交易参与方，包括用户、商户、账户、设备、支付渠道、IP、收货地址等对象。
- 模拟正常交易流程，包括注册、登录、绑卡、支付、退款、转账、提现等典型行为。
- 模拟欺诈交易流程，包括盗刷、薅羊毛、团伙套利、异常频次交易、异地异设备交易、夜间集中交易等场景。
- 支持跨时段行为生成，体现用户长期行为变化，而不是只生成单笔孤立交易。
- 支持跨渠道行为生成，例如 App、网页、小程序、线下扫码等不同入口。
- 给每条交易或事件打标签，明确正常、可疑、欺诈，以及欺诈类型。

建议产出：

- 一套交易仿真数据生成方案说明。
- 一份字段字典，写清楚每个字段含义、类型、取值范围、是否必填。
- 一批可重复生成的数据脚本或生成器。

### 2.2 设计实时检测链路

需要完成的内容：

- 把仿真数据组织成“流式输入”而不是只做离线表格分类。
- 建立实时处理流程，至少要有“数据输入、特征提取、模型推断、风险输出”四个阶段。
- 对每笔交易在发生时给出风险分数或风险等级。
- 支持动态异常检测，而不是只做固定规则判断。
- 保留日志和检测结果，便于后续分析和答辩展示。

建议产出：

- 实时检测流程图。
- 每笔交易的实时输出示例，包括交易编号、风险分数、风险等级、触发原因。
- 流式处理模块代码或原型。

### 2.3 构建多维特征体系

需要完成的内容：

- 构建用户行为特征，例如交易频率、交易金额分布、活跃时间段、常用设备、常用地区。
- 构建设备指纹相关特征，例如设备 ID、设备变更频率、同设备关联账户数、浏览器或终端属性。
- 构建交易网络特征，例如账户和商户之间的转账关系、共同设备、共同收货地址、共同 IP。
- 构建用户关联特征，例如社交关联、联系人关联、团伙共享资源关联。
- 构建时序特征，例如短时间高频交易、连续失败后成功、深夜交易突增、金额突变。
- 说明每类特征的来源、构建方式和为什么对识别有帮助。

建议产出：

- 特征清单表。
- 特征工程文档。
- 重要特征可视化结果。

### 2.4 完成识别模型或检测算法

需要完成的内容：

- 选择适合的基线方法，例如规则模型、机器学习分类模型、异常检测模型。
- 如果数据有关系网络，考虑加入图分析或图神经网络方法。
- 区分“已知欺诈识别”和“未知异常发现”两个问题。
- 输出可量化指标，例如准确率、召回率、F1、AUC、误报率、漏报率、延迟。
- 对比至少两种方案，说明为什么最终选择当前方案。

建议产出：

- 基线模型与改进模型的对比实验。
- 评估指标表。
- 误判案例分析。

### 2.5 做自适应学习与模型迭代设计

需要完成的内容：

- 设计新增欺诈样本进入系统后的更新机制。
- 说明模型如何增量学习、周期重训或规则热更新。
- 说明欺诈手法变化时系统如何继续适配。
- 最好能演示“加入新型样本前后”的识别效果变化。

建议产出：

- 模型迭代流程图。
- 新增样本前后效果对比。
- 样本回流与标注机制说明。

### 2.6 做团伙欺诈识别与溯源分析

需要完成的内容：

- 识别多个账户、设备、IP、商户之间的团伙关联。
- 找出共享资源或异常路径，例如多个账户共用同一设备、同一 IP、同一地址。
- 对高风险节点进行链路追踪。
- 展示可解释的证据，而不是只给出一个黑盒分数。

建议产出：

- 关系图谱或资金链路图。
- 团伙识别案例。
- 触发风险的关键证据说明。

### 2.7 完成系统工程设计

需要完成的内容：

- 设计整体系统架构，包括数据生成、消息队列、实时计算、模型服务、结果展示。
- 考虑高并发、低延迟处理。
- 考虑混合云部署或至少给出可迁移的部署设计。
- 设计数据安全措施，例如脱敏、权限控制、日志审计。
- 设计演示界面或监控面板，方便答辩展示。

建议产出：

- 系统架构图。
- 技术选型说明。
- 性能测试方案和测试结果。

### 2.8 准备最终提交与答辩材料

需要完成的内容：

- 整理项目说明文档。
- 准备代码结构说明。
- 准备系统架构图和流程图。
- 准备性能测试和实验对比结果。
- 准备演示视频或现场演示脚本。
- 准备答辩时对“为什么这么设计”的解释。

建议产出：

- 项目报告。
- PPT。
- 演示脚本。
- 样例测试数据与结果截图。

## 3. 需要查找和准备的数据

### 3.1 交易主表数据

至少需要以下字段：

- 交易 ID
- 用户 ID
- 商户 ID
- 账户 ID
- 交易时间
- 交易金额
- 支付方式
- 交易渠道
- 交易状态
- 地理位置
- IP 地址
- 设备 ID
- 风险标签

重点要查找：

- 金融欺诈检测公开数据集
- 支付交易异常检测数据集
- 可用于生成支付行为模式的数据样本

可优先检索的公开数据关键词：

- IEEE-CIS Fraud Detection
- PaySim mobile money fraud simulation
- transaction fraud detection dataset
- synthetic payment fraud dataset

### 3.2 用户画像与行为序列数据

建议准备的数据：

- 用户注册信息
- 历史交易序列
- 登录时间分布
- 常用设备与常用地区
- 历史风险记录
- 账户余额变化

重点要查找：

- 用户行为序列建模相关数据
- 时间序列异常检测相关案例
- 用户画像字段设计参考

### 3.3 设备与环境数据

建议准备的数据：

- 设备 ID
- 设备类型
- 操作系统版本
- 浏览器或客户端版本
- IP 地址
- 网络环境
- 设备切换记录

重点要查找：

- 设备指纹常见字段
- 终端风控特征设计资料
- 异常设备切换识别案例

### 3.4 关系网络数据

建议准备的数据：

- 用户与设备关联
- 用户与商户关联
- 账户之间转账关系
- 用户与地址关联
- 用户与联系人关联
- 用户与 IP 关联

重点要查找：

- 图风控或反欺诈图谱资料
- 团伙欺诈识别案例
- 图神经网络在风控中的应用资料

### 3.5 欺诈场景与规则样本

建议整理的场景：

- 盗刷
- 短时高频交易
- 异地异设备登录后交易
- 新设备大额交易
- 多账户共用设备
- 多账户向同一商户集中交易
- 小额试探后大额支付
- 夜间异常活跃

重点要查找：

- 金融诈骗典型模式
- 支付风控常见规则
- 欺诈样本标签定义方式

### 3.6 评估与性能测试数据

建议准备的数据：

- 训练集、验证集、测试集划分方案
- 正负样本比例说明
- 新型欺诈样本单独测试集
- 高并发压力测试数据
- 不同时间窗口下的延迟测试数据

重点要查找：

- 风控模型常用评估指标
- 实时系统吞吐与延迟测试方法
- 类不平衡问题处理方法

## 4. 推荐的技术模块拆分

可以按下面几个模块分工：

- 数据仿真模块：负责生成正常交易和欺诈交易。
- 流式处理模块：负责接收和处理交易流。
- 特征工程模块：负责构建用户、设备、关系、时序特征。
- 风险识别模块：负责分类、异常检测、风险评分。
- 图谱分析模块：负责团伙识别和链路追踪。
- 模型迭代模块：负责样本回流、规则更新、模型更新。
- 可视化展示模块：负责实时监控、告警、结果展示。
- 文档与答辩模块：负责材料整理、PPT、演示脚本。

## 5. 按评分标准倒推的重点

如果想更贴近评分，最优先保证以下四项：

- 仿真环境与实时检测能力：不能只交一个离线模型，最好有交易流模拟和实时输出。
- 流式数据处理框架：哪怕做轻量原型，也尽量体现消息流和实时处理过程。
- 自适应学习与模型迭代能力：需要展示系统不是一次性训练完就结束。
- 多维度特征构建：这是赛题一最明确要求的亮点之一。

在此基础上再去拉高总分：

- 创新性：比如加入图神经网络、团伙挖掘、规则与模型融合。
- 实用性：强调和真实支付风控场景贴近。
- 成熟度：代码结构清晰、模块完整、文档规范。
- 安全性：说明脱敏、权限、传输安全、日志审计。

## 6. 建议你们最先启动的事项

如果准备正式做赛题一，建议优先顺序如下：

1. 先定欺诈场景，明确要模拟哪些风险行为。
2. 再定数据字段，先把数据表结构设计出来。
3. 先做一个最小可运行版本，包括数据生成、风险识别、结果输出。
4. 再补流式处理、图谱分析、模型迭代和可视化。
5. 最后集中整理文档、性能测试和答辩材料。

- 第三方支付风控常见特征
- 金融欺诈检测公开数据集
- 支付交易仿真数据生成方法

- 图神经网络反欺诈案例

- 类不平衡样本处理方法

## 8. 一句话总结

赛题一的本质是：做一个“能模拟交易、能实时判别风险、能解释原因、能持续迭代”的金融反欺诈系统，而不是单独训练一个分类器。
