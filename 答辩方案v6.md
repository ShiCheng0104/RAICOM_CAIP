# 方向一：欺诈交易智能识别答辩方案 v6

## 1. v6 新增内容

v6 在 v5 的 Kafka + Flink 实时检测、微批量优化、多模型对比和 Dashboard 演示能力基础上，补齐赛题中“基于图分析方法实现欺诈链路挖掘、识别团伙式欺诈结构、支持可解释性与溯源分析”的要求。

新增能力如下：

```text
1. 新增异构图欺诈团伙挖掘模块 fraudsim.graph_mining
2. 从共享设备、共享 IP、共享商户、共享收款方、资金转移链路中识别疑似团伙
3. 输出团伙表、实体图风险表、团伙证据 JSONL 和效果摘要
4. 将团伙风险并入 Redis 图画像，Flink 实时作业可随交易流补齐图风险信号
5. Model API 新增图挖掘摘要、团伙列表、实体查询接口
6. 风险理由码新增 fraud_ring_detected / shared_device_ring / shared_ip_cluster 等解释
7. Dashboard 新增“图挖掘”页，展示 Top 风险团伙、链路证据和召回/精度指标
8. 训练侧拆分“基础图统计”和“团伙挖掘特征”，避免图挖掘实验特征扰动主模型
```

## 2. 当前系统完整链路

```text
FP-FraudSim 统一数据集
    -> graph_edges / transaction splits / entity profiles
    -> 图分析挖掘疑似团伙
    -> 生成 fraud_groups / entity_graph_risk / group_evidence
    -> 离线训练 LightGBM / XGBoost / CatBoost / sklearn
    -> FastAPI Model API
    -> Kafka transaction_events
    -> PyFlink 窗口特征 + Redis 画像 + 图风险画像 + 微批量模型调用
    -> Kafka risk_results / alert_events / late_events
    -> Dashboard 实时风控、图链路、团伙挖掘、模型对比、人工反馈
```

## 3. 图分析方法设计

本项目没有直接引入重型 GNN 框架，而是实现了一个可落地、可解释、适合答辩演示的异构图分析方法：

```text
节点：
用户、收款方、商户、设备、IP

边和关系：
用户 -> 收款方
用户 -> 商户
用户 -> 设备
用户 -> IP
用户 -> 用户资金链路
多个用户共享同一设备、IP、商户、收款方形成候选团伙
```

挖掘逻辑：

```text
1. 在训练期历史样本上统计共享设备、IP、商户、收款方资源
2. 根据共享用户数、交易量、金额、历史欺诈种子、场景多样性计算资源风险
3. 将高风险共享资源与相关用户做 union / connected component 聚合
4. 对每个联通团伙计算用户数、资源数、共享设备/IP 数、欺诈种子比例、证据数量
5. 输出 graph_mining_group_risk_score 作为团伙风险分
6. 保存样例用户、样例资源、主导注入团伙、场景分布，供人工溯源
```

该方法的优势是解释性强：每个团伙都能回溯到共享设备、共享 IP、共享商户、共享收款方等证据，而不是只给一个黑盒分数。

## 5. 图挖掘效果评估

本次在 `fp_fraudsim_injected` 数据集上运行图挖掘，使用 `train` 作为历史挖掘样本，使用 `train + valid + test` 中的注入标签评估团伙覆盖能力。

总体输出：

| 指标 | 数值 |
|---|---:|
| 疑似团伙数 | 475 |
| 带团伙风险的实体数 | 160,641 |
| 候选共享资源数 | 56,405 |
| 注入欺诈交易数 | 20,600 |
| 注入团伙真值数 | 2,125 |

广覆盖线索池口径：

| 指标 | 数值 |
|---|---:|
| 注入交易行召回 | 56.69% |
| 注入团伙召回 | 77.93% |
| 平均团伙纯度 | 69.91% |
| 标记行数 | 486,350 |

高置信团伙口径，阈值 `graph_mining_group_risk_score >= 0.65`：

| 指标 | 数值 |
|---|---:|
| 标记交易行数 | 4,018 |
| 标记注入欺诈交易数 | 3,966 |
| 注入样本口径精度 | 98.71% |
| 注入交易行召回 | 19.25% |
| 注入团伙召回 | 17.69% |

解释：

```text
低阈值适合做线索发现：召回高，但会带入大量正常/非注入交易，需要人工研判或模型二次排序。
高阈值适合做重点告警：精度高，但召回较低，适合作为团伙风险解释和高危案件池。
```

## 6. 与 LightGBM 主模型的关系

v6 对图特征做了拆分：

```text
基础图统计特征：
实体度数、欺诈边数量、欺诈边比例
默认进入 LightGBM 主模型

团伙挖掘特征：
graph_mining_group_risk_score、团伙用户数、共享设备/IP 数、证据数量等
默认作为旁路解释和 Redis 实时画像
可通过 --with-graph-mining-features 显式进入训练实验
```

本次训练对比：

| 模型版本 | 特征数 | 图统计 | 团伙挖掘入模 | PR-AUC | ROC-AUC | F1 |
|---|---:|---|---|---:|---:|---:|
| `lightgbm/latest` | 67 | 是 | 否 | 0.9727 | 0.9949 | 0.9169 |
| `lightgbm_rebuilt_graph/latest` | 67 | 是 | 否 | 0.9727 | 0.9949 | 0.9169 |
| `lightgbm_graph/latest` | 83 | 是 | 是 | 0.8518 | 0.9645 | 0.7393 |
| `lightgbm_rebuilt_baseline/latest` | 60 | 否 | 否 | 0.9198 | 0.9833 | 0.8497 |

结论：

```text
基础图统计显著提升主模型效果。
团伙挖掘特征直接入模没有提升，反而扰动主模型。
因此当前最稳妥的方案是：
主模型继续使用基础图统计；
团伙挖掘作为旁路图分析、可解释溯源、高危团伙告警和人工研判入口；
后续如要入模，应做更细的时间切分、特征选择和阈值校准。
```

## 7. 实时链路中的体现

```text

```

其中包含：


Flink 实时作业在处理交易时会按 `payer_id / payee_id / merchant_id / device_id / ip_id` 读取图画像，并把结果写入 `graph_features`。

## 8. Dashboard 体现

Dashboard 新增“图挖掘”页：


本轮继续增强后，图挖掘页不再只展示团伙摘要，而是点击团伙后调用后端详情接口，返回完整解释包：

```text
group
evidence
explanations

```

其中 `explanations` 包含：

| 字段 | 含义 |
|---|---|
| `risk_level` | 团伙级风险等级：high / medium / watch |
| `explanation_codes` | 团伙理由码，如共享设备环、共享 IP 聚集、资金转移链 |
| `explanation_text` | 面向人工研判的自然语言摘要 |
| `evidence_summary` | 按证据类型汇总，并列出 Top 风险资源 |

新增团伙级理由码：

| 理由码 | 说明 |
|---|---|
| `high_confidence_ring` | 团伙风险分高，进入高置信研判池 |
| `shared_device_ring` | 多个用户共享高风险设备 |
| `shared_ip_cluster` | 多个用户共享高风险 IP |
| `merchant_cashout_cluster` | 商户侧存在聚集式套现或高风险交易 |
| `shared_payee_chain` | 多用户指向同一收款方或资金链路 |
| `fraud_seed_propagation` | 团伙内存在多个历史欺诈种子 |
| `multi_scenario_campaign` | 同一团伙覆盖多个欺诈场景 |
| `fund_transfer_chain` | 出现 mule account / money laundering 等资金链路场景 |

`subgraph` 是前端可直接绘制的子图 JSON：

```text
nodes:
  团伙中心节点
  风险节点
  样例用户节点
  共享资源节点：设备、IP、商户、收款方

edges:
  用户 -> 团伙：member
  团伙 -> 资源：shared_device / shared_ip / shared_merchant / shared_payee
  团伙 -> 风险：risk_score
```

## 11. 当前边界与后续规划

当前已经补齐“图分析方法”的最小落地版本，但它仍不是完整 GNN 系统。

仍可增强：

```text
1. 引入 GraphSAGE / GAT / R-GCN，在异构图上学习节点 embedding
2. 将图挖掘改成严格时间切分，避免任何未来窗口信息进入历史画像
3. 将高置信团伙写入单独 Kafka topic，例如 fraud_ring_alerts
4. 引入图数据库或向量索引，支持 Dashboard 多跳交互查询
5. 对不同欺诈场景分别建模：洗钱链路、薅羊毛、盗号转账、商户套现
6. 将人工反馈回写团伙风险，做团伙级自适应更新
```

答辩建议表述：

> 本系统已经实现了可解释的图分析欺诈链路挖掘。它不是简单把交易做成一张展示图，而是从共享设备、IP、商户、收款方和资金链路中挖掘疑似团伙，输出团伙风险分和证据链。主模型继续使用稳定的基础图统计保证准确率；团伙挖掘作为旁路分析支持高危团伙发现、人工研判和溯源解释。
